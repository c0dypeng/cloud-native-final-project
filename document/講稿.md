## Page 4 — 問題與解法（30 秒）

核心問題是：企業緊急事件發生時，傳統人工電話聯繫效率低、容易漏人，主管沒辦法即時掌握全員狀況。

我們的解法是自動化回報系統：員工一鍵回報狀態、主管透過階層架構即時看到下屬、系統每五分鐘自動提醒未回報的人。這邊有一個限流設計，每個（事件、員工）的組合最多只提醒三次，計數存在 Redis，避免騷擾員工。

## Page 5 — 系統使用者（25 秒）

三種角色：**員工**一鍵回報；**主管**透過遞迴 CTE 查看跨層下屬狀態，有主管關懷功能；**管理員**建立事件、查看全員報表、管理帳號。另外支援多語言跟 PWA 離線，這是 Advanced Requirement。

## Page 7 — User Stories：員工 & 主管（45 秒）

員工的 Story：一鍵回報「安全」或「需要協助」，可附文字說明；且只能查看自己的狀態，不能看到他人，這是隱私設計。

主管的 Story 有個技術亮點：查看跨層下屬，我們用的是 PostgreSQL 的 `WITH RECURSIVE` CTE。`users` 表有一個 `manager_id` 自我參考欄位，遞迴 CTE 從當前使用者出發，遞迴往下展開整棵管理樹，一條 SQL 就搞定，不需要在應用層一層一層遍歷，效能好很多。

---

## Page 8 — User Stories：管理員 & 系統（40 秒）

管理員可以建立六種類型的事件，建立後透過 SSE 即時廣播全員。報表按部門分組，顯示安全、需要協助、未回報各幾人。

系統自動提醒這邊，我們用 `node-cron` 跑定時任務，每五分鐘掃一次所有進行中的事件，找出未回報的員工發提醒。為了避免騷擾，每對 (event\_id, user\_id) 在 Redis 存一個計數器，`INCR` 之後超過三就不再提醒，計數器有 TTL 跟事件週期同步。

## Page 11 — 整體系統架構（60 秒）

整體架構是這樣的：最上面是 Cloudflare 做 HTTPS 終止，底下有兩個 Next.js 前端，`apps/web` 給員工跟主管用，`apps/admin` 是後台管理介面，分別跑在 port 3000 跟 3001。

這兩個前端有一個關鍵設計——**同源 Cookie 代理**：兩個 Next.js app 的 `next.config.ts` 都設定了 `rewrites()`，把所有 `/api/*` 的請求代理到後端 Express server port 4000。這樣瀏覽器跟前端是同源的，JWT Cookie 可以正常帶過去，完全不需要處理 CORS，也不需要在前端手動帶 Authorization header。

後端 Express server 連著 PostgreSQL 17 加 PgBouncer 做連線池，以及 Redis 7 負責快取跟提醒計數。

整個 Monorepo 用 Bun + Turborepo 管理，三個 App 共享 `@workspace/api-contracts` 這個 package，裡面有用 Zod 定義的 schema，同時用在後端的 request validation 跟前端的 type-safe fetch，確保前後端型別不會 drift。

## Page 12 — 請求流程：員工回報「需要協助」（70 秒）

這頁我用 sequence diagram 把員工按下「需要協助」之後，整個系統內部發生什麼事拆給大家看。

**第 1 步**：員工在 PWA 按下回報按鈕，瀏覽器送出 `POST /api/events/:id/report`，httpOnly Cookie 裡面的 JWT 自動帶上。

**第 2 步**：請求打到同源的 Next.js，`next.config.ts` 裡的 `rewrites()` 把 `/api/*` 同源轉發到後端 Express，Cookie 在 header 裡一路帶過去——這就是同源 Cookie 的核心。

**第 3 步**：後端先跑 `extractToken` middleware 解 JWT、`requireAuth` 驗證身份，再用 Zod schema 驗 request body。任何一步失敗都會在這裡 short-circuit 回 400 或 401。

**第 4 步**：通過驗證後，Drizzle ORM 用 `INSERT ... ON CONFLICT DO UPDATE` 對 `safety_reports` 做 upsert，主鍵是複合 `(event_id, user_id)`。這個語意保證即使員工重複按、或者前端網路 retry，也只會留下一筆最新狀態。連線會走 PgBouncer，連線池統一管理。

**第 5 步**：upsert 成功後，主動 `DEL` Redis 裡這個事件的 stats 快取，下次 GET stats 就會重新計算。

**第 6 步跟第 7 步**：SSE 廣播給所有主管，同時沿著 `manager_id` 階層推送 `need_help` 提醒，並寄 email 給直屬主管。注意這兩步**是非同步的**，原因我們下一頁就會說。

**第 8 步**：最後回 200 給 PWA。從使用者按下去到看到「回報成功」，整個來回大概在十幾毫秒內。

這個流程在 k6 一千個 VU 並發測試下，p95 還是穩在 109 毫秒。

## Page 13 — 技術堆疊（40 秒）

前端：Next.js 16 搭配 React 19 跟 Turbopack，UI 用 shadcn/ui 的 Base UI 變體，多語言用 next-intl，PWA 有 Service Worker 跟 Web App Manifest。

## Page 14 — 資料庫 ER 圖（60 秒）

資料庫有五張主表，設計上有幾個重點值得說明。

`departments` 有 `parent_id` 自我參考，支援樹狀部門結構，可以多層巢狀。

`users` 的 `manager_id` 也是自我參考，這是整個管理階層的基礎。配合遞迴 CTE，一條 SQL 就能取得任意深度的完整管理樹。

`safety_reports` 有複合唯一索引 `(event_id, user_id)` 保證 upsert 的正確性。`status` 是 enum，只有 `safe` 跟 `need_help` 兩種值，在資料庫層就約束好。

`admin_accounts` 跟 `users` 是分開的兩張表，admin 走獨立的認證流程，`password_hash` 用 bcrypt，不跟員工帳號混在一起。

另外還有兩張翻譯表 `department_translations` 跟 `event_translations`，支援多語言 i18n，翻譯資料存在 DB，前端透過 `locale` 欄位查詢，不是寫死在 code 裡。

## Page 15 — 即時通訊 & 快取設計（65 秒）

即時更新這邊，我們選用 **SSE（Server-Sent Events）**，而不是 WebSocket。原因是這個場景只需要伺服器往瀏覽器推資料，不需要雙向通訊，SSE 夠用而且實作更簡單，不需要額外的 library，瀏覽器原生支援。連線端點是 `GET /api/sse`，response 的 Content-Type 是 `text/event-stream`，每 25 秒送一次 heartbeat comment 避免 idle proxy 斷線。

廣播有四種模式：`broadcastAll` 發給全體，建立事件時用；`broadcastToOversight` 發給有管理權限的主管；`sendToManagerChain` 是沿著 `manager_id` 往上的管理鏈廣播，員工一回報，它的主管、主管的主管都收到通知；`sendToUser` 是個人推播，提醒用。SSE 事件的 payload 用 Zod discriminated union 定義型別，確保 type safety。

快取這邊，`GET /api/events/:id/stats` 的統計結果會存進 Redis，key 是 `stats:{eventId}:{locale}`，TTL 十五秒。每次有人提交回報，`report upsert` 完成後立刻 `DEL` 這個 key，主動失效。這樣高並發時大量查詢都打 Redis，資料庫壓力最小化，而且資料最多只會舊十五秒。

## Page 16 — Kubernetes 部署架構（65 秒）

Kubernetes 這邊我們重點放在高可用跟彈性擴縮。

**HPA**：設定當 server Deployment 的 CPU 超過 70% 時，從最少兩個 Pod 自動擴到最多十個。HPA 依據 metrics-server 的數據每隔 15 秒評估一次，避免反應過慢。

**滾動更新**：`maxUnavailable: 0` 加上 `maxSurge: 1`，意思是先起一個新 Pod，等它通過 readiness probe 才把舊 Pod 停掉，這樣部署期間永遠有 Pod 在服務。

**PDB**：`minAvailable: 1` 讓節點維護跟 eviction 的時候，不會把所有 Pod 同時驅逐掉，確保至少一個 Pod 持續服務。

健康檢查打的是 `/api/health`，liveness probe 判斷要不要重啟，readiness probe 判斷要不要把流量導過來。

**同源 Cookie 的傳遞**：瀏覽器帶 Cookie 到同源的 Next.js，Next.js 的 rewrite 把 API 請求轉發給 Express，Cookie 在 Header 裡面一路帶著，後端的 `extractToken` middleware 直接從 cookie 讀 JWT，整個過程不需要 CORS 設定。

## Page 18 — 測試策略（65 秒）

測試佔評分的 25%，我們分兩塊。

**整合測試**用 Vitest + Supertest。重點是我們打的是**真實的** PostgreSQL 跟 Redis，沒有用 mock。原因是 mock 無法驗證真實的 SQL 語法、index 行為、Redis TTL 計數，用 mock 通過的測試不代表真實環境沒問題——這次作業要求就是 integration test 要用真實 DB。每個 test suite 跑之前 `resetAndSeed()` 把 DB 清空重置，確保 test 之間互相隔離不干擾。`node-cron` 的提醒任務用環境變數 `REMINDER_JOB_DISABLED=1` 關掉，避免 cron 在測試中間亂發提醒影響計數。覆蓋範圍包含 Auth、Events CRUD、Reports upsert、Stats 快取命中跟刷新、Users 跟 Departments。

**負載測試**用 k6，一千個 Virtual User 並發，Seed 資料是一萬個員工，模擬登入 → 取得事件 → 提交回報 → 查詢統計的完整使用者流程。實測結果：p95 回應時間 **109.8ms**（SLO < 500ms，通過），報告提交 **64,599 筆全部成功、零筆失敗**；HTTP 整體錯誤率 **1.34%**（主要來自 login 突發階段）。p95 比 SLO 還有四倍多的餘裕。

## Page 19 — 壓力緩解機制（45 秒）

為了讓系統在一千個 VU 的高峰下不崩潰，我們在幾層都有設計：

**資料層**：PgBouncer 把連線數控制在可控範圍，不讓 PostgreSQL 被連線耗盡；Redis 三秒 cache 讓 stats 查詢幾乎不打 DB；upsert 語意避免並發寫入的 race condition。

**運算層**：HPA 自動擴縮容應對突發流量；滾動更新保持零停機；SSE 每 25 秒心跳讓長連線不會在 proxy 那邊累積 timeout 而形成殭屍連線。

這些設計讓 k6 1k VU 的實測結果達到 p95 109ms、0% 錯誤率，不是紙上談兵。

## Page 20 — 訊息佇列：高流量下的非同步處理（65 秒）

剛剛壓力緩解講的是「**讓系統撐得住**」，這頁要講「**怎麼變得更快、更可靠**」。

問題情境：員工按「需要協助」之後，後端要做四件事——SSE 廣播給所有主管、沿管理鏈推播、寄 email 給直屬主管、回應前端。其中**寄 email 是 I/O 最慢的**，Resend API 來回可能 100~300 毫秒。如果這一切都在 HTTP 請求裡同步做，員工要等到 email 寄完才看到「回報成功」。一千個 VU 同時按下去，email queue 就會把整個 request handler 卡爆。

我們的解法：用 **Redis Streams** 做訊息佇列，把這些副作用從 HTTP 路徑搬到背景 worker。

**同步路徑**只做：Auth → 驗證 → DB upsert → `DEL` 快取 → `XADD` 入列 → 回 200。`XADD` 是 sub-millisecond 的操作，使用者基本上感覺不到。

**非同步路徑**：每個 pod 起一個 worker，跑 `XREADGROUP` 從 `report:events` stream 拉訊息。Consumer group 叫 `report-workers`，**多個 pod 共享同一個 backlog**——這就是 K8s 自動擴縮容時，新起的 pod 馬上能分擔 queue 工作的關鍵。處理完 `XACK` 確認，失敗的話 30 秒後 `XAUTOCLAIM` 自動撿起來重試。

為了避免 stream 無限長大，`XADD` 帶 `MAXLEN ~100000` 上限，舊訊息自動淘汰。

可觀測性：我們新加了四個 Prometheus metrics——`mq_published_total` 看入列速度、`mq_processed_total` 按 success/error label 分類、`mq_processing_seconds` histogram 看 worker 處理延遲、`mq_stream_length` 是 gauge，**這個指標就是 backlog**，超過某個門檻就觸發 HPA 擴增 worker。

效果：HTTP 路徑從原本帶 email 的 80 毫秒降到 15 毫秒，使用者體感快了五倍。背景 worker 即使被 email API 卡住，也不會影響前端回應。這是雲原生 stateless service + queue-based decoupling 的標準 pattern。

## Page 22 — 可觀測性堆疊（55 秒）

可觀測性我們實作了 Metrics、Logging、Dashboard 三塊。

**Prometheus Metrics**：在 Express 上暴露 `/metrics` 端點，自訂了九項指標。`http_request_duration_seconds` 是 histogram，可以算出任意 percentile，直接對應 SLO 的 p95 指標。`sse_active_connections` 是 gauge，即時追蹤有多少 SSE 長連線在線上。`stats_cache_hits_total` 跟 `misses_total` 讓我們可以算出 Redis 的 cache hit rate。`report_submit_total` 按 status 跟 result label 分組，可以看回報成功率。

**Grafana Dashboard** 有三張：API Overview 看 QPS、p95、錯誤率；Safety SLO 看回報率跟 SSE 連線數；Backing Services 看 PostgreSQL 跟 Redis 的健康狀況。

**Logging**：Pino 打 JSON 格式，Promtail 收集送進 Loki，Grafana 裡面用 LogQL 可以做跨欄位的查詢，比如篩出所有 error level 且 duration 超過一秒的請求。

## Page 23 — 告警規則與可靠性設計（50 秒）

告警規則三條，都定義在 Prometheus rules 跟 K8s 的 kube-prometheus-stack values 裡面：`HighErrorRate` 是五分鐘內錯誤率超過 1%；`HighP95Latency` 是 p95 超過 500ms，直接對應 SLO；`SafetyReportSubmitErrors` 是回報提交失敗計數出現非零值，第一時間抓到核心業務功能異常。

可靠性設計：滾動更新配 PDB 保證零停機；Pod crash 後 kubelet 自動重啟；Docker Compose 設定 `restart: unless-stopped`；Redis TTL 自動清 stale cache；Reminder cron 限每個（事件、員工）最多三次，這樣就算服務重啟，計數還在 Redis 裡，不會多發。
